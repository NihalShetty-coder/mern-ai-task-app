# Architecture Document

## 1. System Overview
The AI Task Processing Platform is designed as a scalable, distributed system utilizing the MERN stack (MongoDB, Express, React, Node.js) paired with a Python worker service for asynchronous background processing. The entire architecture is containerized using Docker, orchestrated with Kubernetes (specifically k3s for lightweight deployments), and managed via GitOps using Argo CD.

The core objective of the platform is to allow users to authenticate securely, submit text-based tasks for AI processing (such as uppercase, lowercase, reverse string, and word count), and retrieve the status, logs, and final results asynchronously without blocking the user interface.

## 2. Worker Scaling Strategy
The asynchronous worker architecture is the backbone of the platform's reliability under load. We utilize a Python-based background job processor that connects to a Redis Streams queue.
*   **Consumer Groups:** Workers are deployed as stateless consumers belonging to a single Redis Streams Consumer Group (`workers`). This guarantees that each task generated by the backend API is delivered to and processed by exactly one worker replica.
*   **Horizontal Pod Autoscaler (HPA):** In Kubernetes, the worker Deployment is managed by an HPA. It monitors CPU and Memory utilization. If average CPU utilization exceeds 70%, the HPA automatically spins up additional worker replicas (e.g., from 3 up to 12 in the production environment).
*   **Linear Scalability:** Because tasks are independent and idempotent (a task takes an input text and an operation and produces a deterministic output), adding more workers linearly increases the processing throughput of the system.

## 3. Handling High Task Volume (100k tasks/day)
Processing 100,000 tasks per day averages to roughly 1.16 tasks per second. However, system loads are rarely uniform and tend to experience sudden spikes or bursts.
*   **API Decoupling:** The Node.js/Express API does not process tasks. It simply validates the incoming request, inserts a `pending` record into MongoDB, and immediately pushes a message to Redis Streams. This ensures the API remains highly responsive (sub-100ms response times) even during massive traffic spikes.
*   **Redis Backpressure:** Redis Streams acts as a durable shock absorber. If 10,000 tasks are submitted in a single minute, they queue safely in Redis. The workers consume them at their maximum sustained rate without overwhelming the database or crashing.
*   **Stateless Scaling:** As the queue grows, the workers will consume more CPU, triggering the HPA to scale them out horizontally to burn down the backlog.

## 4. Database Indexing Strategy
MongoDB is used as the primary data store for both User accounts and Task metadata/results. Proper indexing is critical to prevent full collection scans as the database grows to millions of tasks.
*   **Users Collection:** A unique ascending index on `{ email: 1 }` ensures fast login lookups and prevents duplicate registrations.
*   **Tasks Collection - User Queries:** A compound index on `{ userId: 1, createdAt: -1 }` is implemented. This optimizes the frontend dashboard query where a user retrieves their historical tasks sorted by the most recent first.
*   **Tasks Collection - Status Tracking:** An index on `{ status: 1, updatedAt: -1 }` is applied. This allows administrative dashboards or background chron jobs to quickly query for stuck tasks (e.g., tasks in a `running` state for too long) to re-queue them.

## 5. Handling Redis Failure
Redis Streams is critical for task queuing. For local/staging (k3s) environments, Redis is deployed as a single pod with Append-Only File (AOF) persistence enabled via persistent volume claims to prevent data loss across pod restarts.
*   **API Resilience:** If the Redis connection fails, the API will fail to queue the job and will return a 503 error to the client, preventing silent failures.
*   **Worker Resilience:** The Python worker is wrapped in an infinite retry loop with exponential backoff. If Redis goes down, the worker will safely wait and automatically reconnect when it comes back online.
*   **Message Durability:** Because we use Redis Consumer Groups, tasks are not deleted from the stream until the worker explicitly calls `XACK`. If a worker pod crashes mid-processing, the task remains in the Pending Entries List (PEL) and can be claimed by another worker after a timeout.

## 6. Staging and Production Environments
The platform separates infrastructure code from application code via a dedicated `mern-ai-task-infra` repository, following GitOps best practices.
*   **Kustomize:** The infra repository uses Kustomize overlays. A `base` configuration defines the core deployments, while `staging` and `prod` overlays apply specific resource limits, replicas, and ingress routing rules.
*   **Staging:** Configured with minimalistic resource requests (e.g., 100m CPU limits) and minimal replicas to save costs while verifying integration.
*   **Production:** Configured with robust resource limits, multiple API replicas, HPA enabled on workers, and strict readiness/liveness probes.
*   **Argo CD:** We maintain two separate Argo CD Applications within the cluster—one watching the `k8s/staging` path and the other watching `k8s/prod`. When CI/CD merges a new Docker image tag into the infra repository, Argo CD automatically detects the drift and syncs the changes to the respective Kubernetes namespace.
